# Leçons — ComRoster

Format : `[date] | ce qui a mal tourné | règle pour l'éviter`

[2026-06-19] | pytest ne trouvait pas le module `comroster` (ModuleNotFoundError dans conftest) | Tout projet Python à la racine sans packaging : créer un `pyproject.toml` avec `[tool.pytest.ini_options] pythonpath = ["."]` dès P0, sinon les imports échouent.
[2026-06-19] | TemplateNotFound : Flask cherche `comroster/templates/` mais les templates sont à la racine du projet | Quand templates/ et static/ sont hors du package, instancier `Flask(__name__, template_folder="../templates", static_folder="../static")`.
[2026-06-19] | Smoke test HTTP local : login/publish renvoyaient 400 (CSRF) car `SESSION_COOKIE_SECURE=True` empêche le cookie de session de circuler sur HTTP non-TLS | Comportement CORRECT en prod (HTTPS). Pour tester en local sans TLS, lancer avec `FLASK_DEBUG=true` (désactive SECURE) ou passer par le test client pytest (CSRF off).
[2026-06-20] | Layout app-shell (header/footer position:fixed + conteneur central flex:1 overflow:auto) : scroll cassé sur admin ET auto-scroll JS inactif sur display | Cause racine commune : `body` avec `min-height:100vh` mais pas `height:100vh`. Le conteneur central n'est jamais borné → pas d'overflow interne, c'est la fenêtre qui scrolle. TOUJOURS borner `body` (ou la page) à `height:100vh; overflow:hidden` pour que les enfants `flex:1; overflow:auto` scrollent en interne.
[2026-06-21] | Élément togglé par l'attribut `hidden` mais avec une règle CSS `display:flex/inline-flex` → l'attribut hidden est écrasé, l'élément reste affiché (ex. assistant+dashboard+étapes tous visibles, barre de sélection collée). Erreur commise 2 fois. | RÈGLE D'OR : ajouter une fois pour toutes `[hidden] { display: none !important; }` au reset CSS. Ne jamais poser un `display` inconditionnel sur un élément piloté par `hidden` (ou utiliser `:not([hidden])`).
[2026-06-22] | Pastilles temps réel (état antenne) ne disparaissaient pas à la déconnexion sans rafraîchir : elles dépendaient d'un poll périodique (5 s) jamais déclenché par l'action utilisateur. | Tout état d'UI alimenté par un polling doit être rafraîchi IMMÉDIATEMENT après l'action qui le modifie (connect/disconnect/import) — appeler la fonction de poll juste après l'action, sans attendre le prochain tick. Ne jamais compter sur l'intervalle seul pour refléter une action explicite.
[2026-06-22] | Un test existant imposait « échouer bruyamment » sur fichier d'état corrompu (load_draft devait lever). Pour un produit livré en boîtier, ça brickerait le client. | Politique APPLIANCE = fail-safe, pas fail-loud : récupérer (sauvegarde .bak puis état vide) AU LIEU de planter, mais TOUJOURS journaliser (warning/error) pour ne pas masquer. Quand on change une politique, mettre à jour le test qui encodait l'ancien contrat plutôt que le contourner.
[2026-07-06] | Le recover permettait un mot de passe vide : la règle de longueur n'existait que sur le chemin setup | Une politique de validation (longueur mdp, format…) doit être appliquée sur TOUS les chemins qui écrivent la même donnée (setup + recover + change). Avant de valider, lister les chemins d'écriture.
[2026-07-06] | Rate-limit par IP inopérant derrière nginx : get_remote_address voit 127.0.0.1 pour tout le monde (compteur partagé → un attaquant verrouille le login de tous) | Toute logique basée sur l'IP client doit prévoir le cas reverse-proxy : ProxyFix opt-in par env var (jamais actif par défaut, sinon spoofable en direct) + le documenter dans la conf du proxy.
[2026-07-06] | `except Exception: return "DISPLAY OK"` (vestige de dev) masquait tout bug de template en prod | Le fail-safe appliance s'applique aux DONNÉES externes (fichiers corrompus, réseau), JAMAIS aux bugs de code : pas de catch-all autour d'un render/logique interne. Un vestige de debug qui avale les exceptions doit sauter avant livraison.
[2026-07-06] | Chaque flux SSE occupe un thread gthread en continu : 8 threads = 8 displays et le serveur entier ne répond plus (même /healthz) | Pour tout endpoint streaming sur un pool de threads borné : imposer un cap applicatif (503 + Retry-After) STRICTEMENT inférieur au nombre de threads, et vérifier que le client sait retenter après une erreur HTTP.
[2026-07-06] | Le lock d'écriture atomique ne protégeait pas le cycle load→mutate→save : deux PATCH simultanés se perdaient (last-write-wins) | L'atomicité d'écriture ≠ l'atomicité de transaction. Tout read-modify-write partagé doit être sérialisé sur toute sa durée (décorateur lock au niveau handler), pas seulement sur l'écriture finale.
